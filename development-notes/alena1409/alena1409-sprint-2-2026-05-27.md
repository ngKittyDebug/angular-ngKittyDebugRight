## Problems & Solutions:

Проблемы, которые я не успела зафиксировать в дневнике, когда правила страницу логина, но они были ранее и о них стоит написать тоже:

**Проблема 1.**

В темплейте появились длинные «макароны» из путей, например:

`facade.loginForm.controls.email.hasError('required')`

Решением стало создать локальную переменную через `@let` и избавиться от этой путевой лапши.

```
@let emailControl = facade.loginForm.controls.email;

emailControl.errors?.['required']
```

Код стал заметно чище и читаемее. Начинаю понимать, насколько важно упрощать шаблоны, потому что конструкции быстро становятся плохо поддерживаемыми.

**Проблема 2.**

Мой facade-сервис создавался сразу при инициализации приложения из-за:

 `providedIn: 'root'`

Даша предложила подумать над тем, насколько вообще правильно провайдить этот сервис глобально.

После того как подробнее почитала про DI и области видимости сервисов, поняла, что замечание действительно очень уместное. Сейчас нет необходимости создавать facade еще до того, как пользователь вообще перешел на страницу логина.

В итоге начала лучше понимать, что сервисы тоже стоит загружать осознанно, а не просто кидать все в root.

Это чем-то похоже на `lazy loading` страниц: нет смысла загружать то, что пока не нужно пользователю. У нас даже переводы отдельных страниц подгружаются лениво, и это тоже помогает не тащить лишнее в initial bundle.

Получается, лень в программировании иногда действительно полезна :)

Что происходило с гвардами:

**Проблема 3.**

Выбор между типом гварда  `CanActivateFn` и `CanMatchFn`.

`CanActivateFn` — Angular уже нашёл маршрут и собирается загрузить компонент, и только тогда гвард проверяет можно ли войти. Если `true` — переход осуществляется, если нет — работает по прописанному сценарию, в моём случае переброс на страницу логина.

`CanMatchFn` — оказался более гибким, первый раз про него читала и не знала про него. В чём его преимущества, как я поняла: он не загружает компонент до того как пройдёт проверка, т.е. мы не засоряем наше приложение лишними загрузками. Это плюс для производительности. Также этот способ будет полезен в случае когда у нас путь называется одинаково, а отобразить надо разные страницы — например разные страницы профиля у админа и у обычного юзера. Если бы такой вариант был реализован через `CanActivateFn`, не сработало бы, так как Angular нашёл бы первый маршрут по пути и до второго никогда не дошёл бы.

```
{
    path: 'profile',
    canMatch: [adminGuard],
    component: AdminProfile
  },
  {
    path: 'prifile,
    canMatch: [userGuard],
    component: UserProfile
  },
  ```

Решение — использовать `CanMatchFn`, так как он более гибкий и улучшает производительность.

**Проблема 4.**

Тесты тесты тесты...
Не умею я писать эти тесты, хорошо тут понятна была сама логика: залогинен — пускает, не залогинен — не пускает на страницу. Пол дела сделано.

Создала сначала мок для теста сама:

```
import { signal } from '@angular/core';

export const authServiceMock = {
  isAuthenticated: signal(false),
};
```

Оказалось, подобный мок уже был - переписала на существующий:
```
export const authServiceMock = {
  user: vi.fn().mockReturnValue(null) as unknown as Signal<null | User>,
  isLoading: vi.fn().mockReturnValue(false) as unknown as Signal<boolean>,
  lastCallbackResult: vi.fn().mockReturnValue(null) as unknown as Signal<CallbackResult | null>,
  isAuthenticated: vi.fn().mockReturnValue(false) as unknown as Signal<boolean>,
  initialize: vi.fn().mockResolvedValue(undefined),
  loadUser: vi.fn().mockResolvedValue(null),
  login: vi.fn().mockResolvedValue(null),
  signup: vi.fn().mockResolvedValue(null),
  logout: vi.fn().mockResolvedValue(undefined),
  loginWithProvider: vi.fn(),
  loginWithGithub: vi.fn(),
  loginWithGoogle: vi.fn(),
  requestPasswordRecovery: vi.fn().mockResolvedValue(undefined),
} as const satisfies MockedObject<Partial<AuthService>>;
```

Использовала `isAuthenticated: vi.fn().mockReturnValue(false) as unknown as Signal<boolean>`, но конечно всё было не гладко - не хотели подтягиваться типы, поэтому пришла к помощи чата, который подсказал, что надо типизировать так:

 ```
 const isAuthenticatedMock = authServiceMock.isAuthenticated as unknown as MockedFunction<() => boolean>;
 ```

Надеюсь, когда-то я сама смогу это всё понимать.

Также в тестах для имитации отработки гварда надо использовать:
- с `CanMatch`: `executeGuard(RouterFixtures.route, RouterFixtures.segments)`
- с `CanActivate`: `executeGuard(RouterFixtures.activatedRouteSnapshotFixture, RouterFixtures.routerStateFixture)`

Для этого был создан файл фикстур:

 ```
 export const RouterFixtures = {
  activatedRouteSnapshotFixture: {} as unknown as ActivatedRouteSnapshot,
  routerStateFixture: {} as unknown as RouterStateSnapshot,
  route: {} as unknown as Route,
  segments: [] as UrlSegment[],
};
```
