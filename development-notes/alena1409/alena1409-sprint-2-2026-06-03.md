## What I learned:

**Как добавить роутинг**

В файле `app.routes.ts` при создании приложения создаётся массив с роутами, куда собственно мы и прописываем наши пути - пишем их объектом.

`router-outlet` - для работы роутинга в шаблоне нужно разместить `<router-outlet />`. Без него компоненты не будут рендериться, даже если роуты корректны.

Примеры:

**1. Обычный роут**

```ts
{
  path: '',
  component: MainComponent,
}
```

В данном случае у нас не указан никакой путь - это взято за главную страницу, компонент просто подгружается в момент, когда открывается приложение.

**2. Роут с провайдерами**

```ts
{
  path: '',
  component: MainComponent,
  providers: [provideTranslocoScope('main'), MainPageFacade],
}
```

Так мы можем провайдить какие-то сервисы, фасады библиотеки непосредственно через роутинг - для того чтобы не подключать что-то глобально и сэкономить на производительности.

Провайдить можно также через `providedIn: 'root'` - тогда компонент или сервис будет виден всему приложению. Также можно провайдить непосредственно в компонент, если нам нужно подгружать что-то только в определённом месте и нет необходимости всему приложению это видеть.

**3. Ленивая загрузка + гвард**

```ts
{
  path: 'login',
  loadComponent: () =>
    import('./features/login/ui/login.component').then((c) => c.LoginComponent),
  providers: [provideTranslocoScope('login'), LoginPageFacade],
  canActivate: [guestGuard],
},
```

Тут мы можем заметить, что у нас подключена ленивая загрузка с помощью `loadComponent` и подключён гвард. Гвард можно настраивать по своему усмотрению. В данном случае если пользователь не авторизован - ему будет доступна страница логина; если авторизован - будет перекидывать на главную или другую заданную страницу в гварде.

**4. Дочерние роуты**

```ts
{
  path: '',
  component: LayoutComponent,
  children: [
    {
      path: '',
      component: MainComponent,
    },
  ],
}
```

Тут продемонстрирована возможность добавлять дочерние пути. Это удобно, когда несколько страниц должны разделять общий лейаут (шапка, сайдбар и т.д.).

**5. `pathMatch: 'full'`**

Если мы укажем `pathMatch: 'full'` - это будет означать, что путь полностью должен соответствовать заявленному. Обычно используется на пустом `path: ''`, чтобы не перехватывать все подпути, чаще всего в паре с редиректом:

```ts
{ path: '', pathMatch: 'full', redirectTo: 'home' }
```

**6. `loadComponent` vs `loadChildren`**

`loadComponent` - лениво загружает один standalone-компонент.
`loadChildren` - лениво загружает целый модуль или файл с массивом роутов (`Routes`). Используется, когда у фичи есть свои внутренние роуты:

```ts
{
  path: 'dashboard',
  loadChildren: () =>
    import('./features/dashboard/dashboard.routes').then((r) => r.DASHBOARD_ROUTES),
}
```

## Problems & Solutions:

**Проблема**

При переходе на страницы приходится ждать несколько секунд, пока загрузится страница.

**Решение**

Даша познакомила нас с такой интересной штукой, как прелоадинг. Что это такое: это предзагрузка страниц в фоновом режиме. Как итог — нам не приходится ждать, когда подгрузится страница.

Я думала, что это напрямую конфликтует с ленивой загрузкой, но оказалось - совершенно нет. Ленивая загрузка не даёт нам подгружать весь код и не раздувает наш бандл, чтобы при первичном заходе на страницу она подгрузилась быстро - это время сокращает нам ленивая загрузка.

А прелоадинг далее в фоновом режиме незаметно для пользователя подгружает нужные страницы. У нас в проекте настроено так, что для гостя подгружаются одни страницы, а для авторизованного пользователя - другие. Очень логично и понятно, особенно когда до этого сам не додумался, а кто-то рассказал - главное уложить это в голове и в будущем использовать тоже.

Подключается прелоадинг в конфигурации роутера, например:

```ts
export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes, withPreloading(PreloadAllModules)),
  ],
};
```

Или с кастомной стратегией (как у нас в проекте - разные страницы для гостя и авторизованного).
