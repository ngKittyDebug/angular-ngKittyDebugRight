# Стайлгайд по тестированию (Angular + Vitest)

## 1. Общие принципы

- Тестируем только **публичный контракт**: публичные методы компонентов, сервисов и фасадов (поведение, видимое снаружи).
- **Один тест — один смысл.** В идеале — один основной `expect`.
- Каждый тест **независим**: порядок выполнения не должен влиять на результат.
- В тестах **нет собственной бизнес-логики** — только подготовка, действие и проверка (**Arrange — Act — Assert**).
- **Комментарии** — только там, где без них неочевидно, _почему_ так настроен тест. Стандартные метки AAA в коде не пишем.
- Группировка по сценариям: **Happy Path**, **Edge Cases**, **Negative Cases** (или русские эквиваленты контекста — см. ниже).
- Файлы тестов: `*.spec.ts` **рядом** с тестируемым файлом.

## 2. Именование

### 2.1. Файлы

```text
user-list.component.spec.ts
user.service.spec.ts
user.facade.spec.ts
```

### 2.2. `describe` — иерархия, на русском, с заглавной буквы

Описывает контекст: от общего к частному.

```typescript
describe('UserListComponent', () => {
  describe('Данные получены', () => {
    it('должен отобразить список пользователей', () => {});
  });
});
```

**Рекомендуемая вложенность:**

| Уровень | Содержание                                                                                                                  |
| ------- | --------------------------------------------------------------------------------------------------------------------------- |
| Внешний | Что тестируется (компонент, сервис, фасад).                                                                                 |
| Второй  | Сценарий: успех / граничные случаи / ошибки (`Happy Path`, `Edge Cases`, `Negative Cases` или предметные русские названия). |
| Третий  | Конкретное состояние или входные данные («Данные получены», «Список пуст», «Ошибка загрузки»).                              |

Пример с английскими метками сценариев:

```typescript
describe('UserListComponent', () => {
  describe('Happy Path', () => {
    describe('Данные получены', () => {
      it('должен отобразить список пользователей', () => {});
    });
  });

  describe('Edge Cases', () => {
    describe('Список пуст', () => {
      it('должен отобразить сообщение о пустом списке', () => {});
    });
  });

  describe('Negative Cases', () => {
    describe('Ошибка загрузки', () => {
      it('должен отобразить сообщение об ошибке', () => {});
    });
  });
});
```

**Варианты сценариев:**

| Блок             | Назначение                                          |
| ---------------- | --------------------------------------------------- |
| `Happy Path`     | Основной успешный сценарий.                         |
| `Edge Cases`     | Границы: пустые данные, максимум длины, нули и т.п. |
| `Negative Cases` | Ошибки, отказы, некорректный ввод.                  |

### 2.3. `it` — на русском, со строчной буквы

Формулировка — ожидаемое **поведение** в данном контексте.

```typescript
it('должен отобразить список пользователей', () => {});
it('должен вызвать saveUser с переданными данными', () => {});
it('должен показать уведомление об ошибке', () => {});
it('не должен вызывать deleteUser для невыбранного пользователя', () => {});
```

Тест на создание сущности по команде проекта:

```typescript
it('должен инициализироваться', () => {});
```

## 3. Структура теста: AAA

Три части разделяются **одной пустой строкой**. Комментарии к блокам AAA не пишем.

```typescript
it('должен вернуть сумму двух чисел', () => {
  const a = 2;
  const b = 3;

  const result = service.sum(a, b);

  expect(result).toBe(5);
});
```

Если блок Arrange неочевиден — допустим **краткий** поясняющий комментарий.

```typescript
// Эмулируем ответ сервера с задержкой для проверки состояния загрузки
userServiceMock.getById.mockReturnValue(of(user).pipe(delay(500)));
```

## 4. Fixture — фиксированные данные

**Fixture** — объект с готовыми тестовыми данными. Предпочтительно **`as const satisfies Тип`** для проверки формы данных на этапе компиляции.

### 4.1. Расположение

```text
feature/
└── fixtures/
    └── user.fixture.ts
```

### 4.2. Паттерн именования и содержимого

```typescript
import { User } from '../models/user.dto';

export const UserFixture = {
  id: 1,
  name: 'Иван',
  role: 'user',
} as const satisfies User;
```

### 4.3. Правила

- Использовать **`as const satisfies`** (или эквивалент по договорённости команды) для связки типа и иммутабельности.
- Не выносить в Fixture данные для **одного** теста; если повтор в **2+** тестах — выносить.
- Папка `fixtures/` — в корне **фичи** (или по соглашению проекта — единообразно везде).

## 5. Mock — подмена зависимости

**Mock** полностью подменяет зависимость (сервис, фасад и т.д.).

### 5.1. Расположение

Рядом с файлом, который подменяется.

```text
feature/
└── services/
    ├── user.service.ts
    ├── user.service.spec.ts
    └── user.service.mock.ts
```

```text
feature/
└── components/
    └── user-list/
        ├── user-list.component.ts
        ├── user-list.component.spec.ts
        └── user-list.component.mock.ts
```

### 5.2. Паттерн создания

Типизация через **`as const satisfies MockedObject<Partial<T>>`** (из `vitest`).

```typescript
import { vi, type MockedObject } from 'vitest';
import { UserService } from './user.service';

export const UserServiceMock = {
  getUsers: vi.fn(),
  getUserById: vi.fn(),
  saveUser: vi.fn(),
  deleteUser: vi.fn(),
} as const satisfies MockedObject<Partial<UserService>>;
```

### 5.3. Правила

- Все подменяемые методы — **`vi.fn()`**.
- Поведение (`mockReturnValue`, `mockImplementation`) задаётся в **Arrange** конкретного теста.
- Если поведение **одинаково** для всех тестов в `describe` — настройка в **`beforeEach`**.

## 6. Spy — наблюдение за реальным объектом

**Spy** оборачивает метод реального экземпляра: проверка вызовов и точечная подмена возврата.

| Выбор    | Когда                                                                 |
| -------- | --------------------------------------------------------------------- |
| **Mock** | Нужно подменить весь сервис целиком, много методов.                   |
| **Spy**  | Нужно проверить или подменить **конкретный** метод реального объекта. |

```typescript
vi.spyOn(service, 'getUsers').mockReturnValue(of([]));
```

### Проверки вызовов

- Количество вызовов — **`toHaveBeenCalledTimes(n)`**.
- Аргументы **конкретного** вызова — **`toHaveBeenNthCalledWith(n, ...)`**.

```typescript
expect(mock.method).toHaveBeenCalledTimes(1);
expect(mock.method).toHaveBeenCalledTimes(0);
expect(mock.method).toHaveBeenNthCalledWith(1, UserFixture.DEFAULT.id);
expect(mock.method).toHaveBeenNthCalledWith(2, UserFixture.ADMIN.id);
```

**Избегать** (слишком неоднозначно для строгого стиля):

- `toHaveBeenCalled()` — заменить на `toHaveBeenCalledTimes(1)`, если ожидается ровно один вызов.
- `toHaveBeenCalledWith(...)` — для первого вызова предпочтительно `toHaveBeenNthCalledWith(1, ...)`.

## 7. TestBed: `configureTestingModule` и переопределения

### 7.1. Первый `beforeEach` — базовая конфигурация

Один **`TestBed.configureTestingModule`** на группу тестов с одинаковыми зависимостями; не копировать блок между тестами без нужды.

```typescript
import { vi, type MockedObject } from 'vitest';

describe('UserListComponent', () => {
  describe('Happy Path', () => {
    describe('Данные получены', () => {
      let component: UserListComponent;
      let fixture: ComponentFixture<UserListComponent>;
      let userFacadeMock: MockedObject<Partial<UserFacade>>;

      beforeEach(async () => {
        // в примере тут, а в проекте распологаем в отдельном файле!
        userFacadeMock = {
          users$: of(UserFixture.LIST),
          loading$: of(false),
          loadUsers: vi.fn(),
          deleteUser: vi.fn(),
        } as const satisfies MockedObject<Partial<UserFacade>>;

        await TestBed.configureTestingModule({
          imports: [UserListComponent],
          providers: [{ provide: UserFacade, useValue: userFacadeMock }],
        }).compileComponents();

        fixture = TestBed.createComponent(UserListComponent);
        component = fixture.componentInstance;
        fixture.detectChanges();
      });

      /* тесты */
    });
  });
});
```

### 7.2. `overrideComponent` / `overrideProvider`

Для **точечной** замены входа/провайдера без дублирования всего `configureTestingModule` — во вложенном `beforeEach` или в начале конкретного `it`.

Пример другого набора данных для того же компонента:

```typescript
describe('UserListComponent', () => {
  describe('Edge Cases', () => {
    describe('Список пуст', () => {
      let fixture: ComponentFixture<UserListComponent>;
      let userFacadeMock: MockedObject<Partial<UserFacade>>;

      beforeEach(async () => {
        userFacadeMock = {
          users$: of([]),
          loading$: of(false),
          loadUsers: vi.fn(),
          deleteUser: vi.fn(),
        } as const satisfies MockedObject<Partial<UserFacade>>;

        await TestBed.configureTestingModule({
          imports: [UserListComponent],
          providers: [{ provide: UserFacade, useValue: userFacadeMock }],
        }).compileComponents();

        fixture = TestBed.createComponent(UserListComponent);
        fixture.detectChanges();
      });

      /* тесты */
    });
  });
});
```

`overrideComponent` — когда нужно подменить метаданные/шаблон **конкретного** компонента (синтаксис зависит от версии Angular — сверяйтесь с актуальной документацией).

### 7.3. Правила TestBed

- **`configureTestingModule`** — один на группу с общей конфигурацией.
- Зависимости — **`useValue`** с mock-объектом.
- **`overrideProvider`** — точечная замена одного провайдера.
- Разные конфигурации → **разные** вложенные `describe` со своим `beforeEach`.

## 8. Асинхронность: `fakeAsync`, `tick`, `flushMicrotasks`, `done`

### 8.1. `fakeAsync` + `tick` — таймеры и синхронизируемые задержки RxJS

```typescript
import { fakeAsync, tick } from '@angular/core/testing';

it('должен установить флаг загрузки с задержкой', fakeAsync(() => {
  component.loadData();

  expect(component.isLoading).toBe(true);

  tick(500);

  expect(component.isLoading).toBe(false);
}));
```

### 8.2. `done` — колбэки и сценарии вне зоны `fakeAsync`

```typescript
it('должен вернуть данные пользователя через колбэк', (done) => {
  service.getUserWithCallback(1, (user) => {
    expect(user).toEqual(UserFixture.DEFAULT);
    done();
  });
});
```

### 8.3. Выбор инструмента

| Ситуация                                               | Инструмент                                                |
| ------------------------------------------------------ | --------------------------------------------------------- |
| `setTimeout` / `setInterval`                           | `fakeAsync` + `tick`                                      |
| Задержки в Observable (`delay`, `debounceTime` и т.п.) | `fakeAsync` + `tick` / `flushMicrotasks` по необходимости |
| Микрозадачи (`Promise`, `queueMicrotask`)              | `fakeAsync` + `flushMicrotasks`                           |
| Колбэки без возможности перевести в `fakeAsync`        | `done`                                                    |
| Реальный HTTP                                          | не ходить в сеть — **mock** сервиса / адаптера            |

## 9. Vitest: практические правила

- **Очистка моков:** `clearMocks: true` включён по умолчанию в `@angular/build:unit-test` — не нужно вручную вызывать `vi.clearAllMocks()` в `afterEach`.
- **`vi.mock` на уровне модуля** — по умолчанию не использовать; предпочтительно явная подмена через **TestBed** и файлы `*.mock.ts` (если команда не приняла отдельное исключение для legacy).
- **Приватные методы** не тестируем напрямую; сложную логику выносим в **хелпер/сервис** с публичным контрактом и тестируем его.
- **`fixture.detectChanges()`** — обычно один раз в `beforeEach`; повторно в `it`, если изменились входы или состояние после действия.
- **Не тестируем фреймворк:** не проверяем «работает ли `@if`», проверяем **условие и данные**, которые вы задаёте приложению.
- **Глобальные API:** `describe`, `it`, `expect`, `beforeEach`, `afterEach` доступны глобально (через `vitest/globals` в `tsconfig.spec.json`). Явный импорт из `vitest` нужен только для `vi`, `MockedObject` и других утилит.

## 10. Чек-лист перед написанием теста

- [ ] Файл: `*.spec.ts` рядом с исходником.
- [ ] Mock: `*.mock.ts` рядом с подменяемым файлом.
- [ ] `describe`: внешний — сущность; далее сценарий; далее контекст.
- [ ] `it`: русский текст, со строчной буквы, описывает поведение.
- [ ] Структура **AAA**, блоки разделены пустой строкой.
- [ ] Комментарий только при неочевидном Arrange.
- [ ] Основной принцип: **один смысл на тест**; несколько `expect` — только если проверяют один результат (или явно разнесены по `it`).
- [ ] Fixture в `fixtures/`, тип через `satisfies`.
- [ ] Mock: `vi.fn()`, типизация `MockedObject<Partial<T>>`.
- [ ] Вызовы: `toHaveBeenCalledTimes`, `toHaveBeenNthCalledWith`.
- [ ] Таймеры и `delay`: `fakeAsync` + `tick` / `flushMicrotasks`.
- [ ] Колбэки без `fakeAsync`: `done`.
- [ ] Один `configureTestingModule` на группу; `overrideComponent` / `overrideProvider` для точечных замен.
- [ ] Тестируется публичное поведение, без привязки к приватной реализации.
