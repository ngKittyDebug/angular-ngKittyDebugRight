# Стайлгайд архитектуры проекта

## Цель

Этот документ фиксирует целевую архитектуру проекта и правила размещения кода в приложении.

Задачи архитектуры:

- сделать структуру проекта предсказуемой;
- отделить бизнес-логику от UI;
- упростить масштабирование фич;
- снизить связанность между частями приложения;
- сделать код удобным для ревью, тестирования и рефакторинга.

## Базовые принципы

- Приложение строится вокруг `feature`-областей.
- Каждая фича изолирует свой UI, state и data access.
- `Page`-компоненты собирают экран, а не содержат тяжелую бизнес-логику.
- Повторно используемая логика поднимается в `shared`.
- Глобальные сущности приложения живут в `core`.
- Зависимости должны быть однонаправленными: инфраструктура не должна зависеть от конкретной фичи.

## Верхнеуровневая структура

Рекомендуемая структура:

```text
src/app/
├── app.config.ts
├── app.routes.ts
├── app.ts
├── core/
├── shared/
└── features/
```

## Назначение верхних слоев

### `app/`

Корень приложения содержит только точку входа и верхнеуровневую оркестрацию:

- bootstrap приложения;
- корневой роутинг;
- провайдеры приложения;
- root-shell.

В `app.ts`, `app.config.ts`, `app.routes.ts` не должно быть бизнес-логики конкретной фичи.

### `core/`

`core` содержит глобальные сущности приложения, существующие в единственном экземпляре.

Сюда относятся:

- root layout;
- глобальные сервисы;
- app-level facade/store;
- interceptors;
- tokens;
- configuration providers;
- навигационные оболочки;
- глобальные UI-обертки, если они относятся ко всему приложению.

Примеры:

- `core/ui/components/layout`
- `core/services`
- `core/tokens`
- `core/interceptors`

`core` не должен зависеть от конкретной feature-области.

### `shared/`

`shared` содержит переиспользуемые сущности, не являющиеся глобальной инфраструктурой и не привязанные к одной фиче.

Сюда могут входить:

- валидаторы;
- гарды;
- общие helper-функции;
- конвертеры общего назначения;
- директивы и пайпы общего уровня;
- общие модели, если они действительно кросс-фичевые;
- переиспользуемые UI-компоненты;
- тестовые моки и фикстуры;
- общие UI-модели;
- презентационные компоненты без привязки к одной фиче.

Примеры:

- `shared/validators`
- `shared/guards`
- `shared/helpers`
- `shared/ui/components`
- `shared/mocks`
- `shared/fixtures`

`shared` не должен знать о конкретных feature-потоках и бизнес-сценариях.

### `features/`

`features` - главный слой прикладной логики.

Каждая feature-область:

- содержит свои страницы;
- владеет своим UI;
- инкапсулирует работу с API;
- содержит свои модели, фасады, моки, хелперы и тесты.

Именно в `features` живет основной продуктовый код.

## Архитектура feature-области

Рекомендуемая структура внутри фичи:

```text
src/app/features/<feature-name>/
├── <feature-name>.routes.ts
├── data/
│   ├── api/
│   │   └── <entity>/
│   │       ├── models/
│   │       ├── services/
│   │       ├── helpers/
│   │       └── mocks/
│   ├── facades/
│   ├── services/
│   ├── models/
│   ├── helpers/
│   └── mocks/
└── ui/
    └── components/
        └── <feature-page>/
            ├── <feature-page>.component.ts
            ├── <feature-page>.component.html
            ├── <feature-page>.component.scss
            ├── <feature-page>.component.spec.ts
            └── <block-name>/
                ├── <block-name>.component.ts
                ├── <block-name>.component.html
                ├── <block-name>.component.scss
                └── <block-name>.component.spec.ts
```

## Ответственность слоев внутри feature

### `data/`

Слой `data` отвечает за данные и бизнес-оркестрацию.

Здесь находятся:

- API services;
- facades;
- state management;
- модели;
- конвертеры;
- helpers;
- mocks.

#### `data/api/`

Этот подслой нужен для интеграции с внешним источником данных.

В него входят:

- сервисы запросов;
- API-модели;
- функции преобразования ответа;
- моки API-сервисов.

Пример:

```text
data/api/space/
├── services/
├── models/
├── helpers/
└── mocks/
```

#### `data/facades/`

Фасад - это точка входа UI в бизнес-логику feature.

Задачи фасада:

- агрегировать данные из нескольких сервисов;
- управлять `signal`-состоянием;
- скрывать детали реализации от компонентов;
- предоставлять понятный публичный API для страницы.

UI должен работать преимущественно через facade, а не напрямую через API-service.

#### `data/services/`

Сервисы feature-уровня содержат бизнес-логику, которая не должна находиться в компоненте.

Примеры:

- orchestration нескольких запросов;
- правила подготовки данных;
- локальные вычисления;
- вспомогательные stateful-сервисы.

### `ui/components`

`ui/components` содержит route-level компоненты страниц и feature-level компоненты верхнего уровня.

Их задача:

- собрать экран из блоков;
- подключить facade;
- передать данные в дочерние компоненты;
- обработать route-level сценарии.

`Page` не должна превращаться в место хранения всей логики приложения.

Если код страницы разрастается, логика выносится в facade или service.

### `ui/components/<page-name>/`

`ui/components/<page-name>` содержит блоки, секции и составные компоненты, используемые внутри конкретной страницы.

Обычно сюда попадают:

- карточки;
- таблицы;
- панели;
- виджеты;
- пустые состояния;
- сообщения об ошибке;
- секции экрана.

Каждый крупный блок интерфейса должен иметь собственную папку.

Если блок используется только внутри одной страницы, он должен жить рядом с этой страницей в `ui/components/<page-name>/`.

Это дает:

- предсказуемую структуру;
- изоляцию шаблона, стилей и тестов;
- понятную точку роста для блока.

## Как собирать страницу

Рекомендуемый поток сборки экрана:

1. Маршрут загружает `page`-компонент.
2. `Page` инжектирует facade.
3. `Page` собирает экран из блоков из `ui/components/<page-name>/`.
4. Компоненты получают данные через `input()` или facade API.
5. Сервисы и facade работают с `data/api`.

Упрощенная схема:

```text
route -> page -> ui components -> facade -> services/api -> backend
```

## Правила зависимостей

Разрешенные зависимости:

- `app` может зависеть от `core`, `shared`, `features`;
- `features` могут зависеть от `core`, `shared`;
- `shared` не должен зависеть от `features`;
- `core` не должен зависеть от конкретных `features`;
- `data` не должен зависеть от `ui`.

Нежелательные зависимости:

- feature A импортирует внутренние файлы feature B;
- `shared` импортирует специфичную бизнес-логику feature;
- `core` начинает хранить продуктовый код одной конкретной страницы;
- компонент напрямую вызывает несколько API-service, минуя facade;
- бизнес-правила остаются в шаблоне или компоненте страницы.

## Когда выносить код выше

Используйте простой критерий:

- если код нужен только одной странице - оставляйте его внутри feature;
- если код нужен нескольким страницам одной feature - поднимайте на уровень feature;
- если код нужен нескольким независимым features - переносите в `shared`;
- если код относится ко всему приложению - переносите в `core`.

## Именование и файловая дисциплина

Архитектура должна идти вместе с едиными правилами именования:

- одна сущность на файл;
- `kebab-case` для файлов и папок;
- для компонентов - отдельная папка на компонент;
- для route-level страницы - отдельная папка в `ui/components`;
- для page-scoped блоков - отдельные папки внутри `ui/components/<page-name>/`;
- для тестов, моков и helper-функций сохраняется то же доменное расположение, что и у основной сущности.

## Практический пример

Пример feature-области `main`:

```text
src/app/features/main/
├── main.routes.ts
├── data/
│   ├── api/
│   │   └── space/
│   │       ├── models/
│   │       ├── services/
│   │       └── mocks/
│   ├── facades/
│   └── helpers/
└── ui/
    └── components/
        └── main-page/
            ├── main-page.component.ts
            ├── main-page.component.html
            ├── main-page.component.scss
            ├── main-page.component.spec.ts
            ├── hero-banner/
            ├── photo-card/
            └── error-state/
```

В таком варианте:

- `main-page` является route-level компонентом страницы;
- `photo-card` и `error-state` являются UI-блоками;
- page-specific блоки лежат в `ui/components/main-page`;
- `SpaceApiService` живет в `data/api/space/services`;
- моки этого сервиса живут рядом, в `data/api/space/mocks`;
- facade связывает API и UI.

## Что считается хорошим признаком архитектуры

- Новый разработчик быстро понимает, куда положить файл.
- Для новой страницы не нужно изменять структуру всего проекта.
- Компоненты остаются небольшими и понятными.
- UI не знает деталей сетевого слоя.
- Переиспользуемые сущности не дублируются между фичами.
- История роста feature хорошо читается по структуре папок.

## Итоговое правило

Архитектура проекта должна быть feature-first:

- глобальное - в `core`;
- общее техническое и переиспользуемое прикладное - в `shared`;
- продуктовая логика - в `features`;
- route-level компонент страницы - в `ui/components`;
- page-specific блоки - в `ui/components/<page-name>`;
- работа с данными и состоянием - в `data`.
