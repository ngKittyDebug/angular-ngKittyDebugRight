# Стайлгайд: нейминг и структура (Angular)

## 1) Компоненты

**Паттерн:** `сущность-тип`

| Тип                 | Шаблон                            | Пример                                                   |
| ------------------- | --------------------------------- | -------------------------------------------------------- |
| Список              | `сущность-list`                   | `user-list`, `order-list`                                |
| Карточка/элемент    | `сущность-card` / `сущность-item` | `user-card`, `order-item`                                |
| Форма               | `сущность-form`                   | `user-form`, `order-form`                                |
| Детали              | `сущность-detail`                 | `user-detail`, `order-detail`                            |
| Диалог              | `сущность-dialog`                 | `confirm-dialog`, `user-dialog`                          |
| Переиспользуемый UI | `назначение-component`            | `button-component`, `modal-component`, `table-component` |

### Размещение page-компонентов и внутренних блоков

- Route-level компонент страницы располагается в `ui/components/<page-name>/`.
- Имя page-компонента подчиняется тому же правилу `сущность-тип`.
- Блоки, используемые только внутри одной страницы, располагаются рядом с ней, внутри `ui/components/<page-name>/`.
- Если компонент начинает переиспользоваться в нескольких страницах одной фичи, его нужно поднимать на уровень выше внутри `ui/components`.
- Если компонент переиспользуется между несколькими фичами, его нужно переносить в `shared/ui/components`.

Пример:

```text
features/main/ui/components/
└── main-page/
    ├── main-page.component.ts
    ├── main-page.component.html
    ├── main-page.component.scss
    ├── hero-banner/
    ├── photo-card/
    └── error-state/
```

## 2) Сервисы

**Паттерн:** `сущность/роль + Service`

| Тип сервиса   | Пример                                |
| ------------- | ------------------------------------- |
| CRUD API      | `UserApiService`, `OrderApiService`   |
| Бизнес-логика | `AuthService`, `PermissionService`    |
| Утилитный     | `DateService`, `NotificationService`  |
| Форма         | `UserFormService`, `OrderFormService` |

Для каждой сложной формы выделяется отдельный form-сервис.

```typescript
@Injectable({ providedIn: 'root' })
export class UserFormService {
  public readonly form = new FormGroup({
    name: new FormControl('', { validators: [Validators.required] }),
    email: new FormControl('', {
      validators: [Validators.required, Validators.email],
    }),
    addresses: new FormArray<UserAddressFormGroup>([]),
  });
}
```

## 3) Реактивные формы: именование контролов

**Паттерн:** `сущность + FormControl / FormGroup / FormArray`

| Тип               | Шаблон           | Пример                                    |
| ----------------- | ---------------- | ----------------------------------------- |
| `FormControl`     | `xxxFormControl` | `nameFormControl`, `emailFormControl`     |
| `FormGroup` (тип) | `xxxFormGroup`   | `addressFormGroup`, `userFormGroup`       |
| `FormArray` (тип) | `xxxFormArray`   | `addressesFormArray`, `itemListFormArray` |
| Экземпляр формы   | `xxxForm`        | `userForm`, `orderForm`                   |

```typescript
export type UserFormGroup = FormGroup<{
  name: FormControl<string>;
  email: FormControl<string>;
  addresses: FormArray<AddressFormGroup>;
}>;
```

```typescript
public readonly userForm = this.userFormService.form;
public readonly nameFormControl = this.userForm.controls.name;
public readonly addressesFormArray = this.userForm.controls.addresses;
```

## 4) Методы

| Назначение          | Префикс                                 | Пример                                        |
| ------------------- | --------------------------------------- | --------------------------------------------- |
| Запрос данных с API | `get` / `fetch`                         | `getUserList`, `fetchOrderById`               |
| Изменение данных    | `save` / `update` / `delete` / `create` | `saveUser`, `updateOrder`, `deleteItem`       |
| Проверка состояния  | `is` / `has` / `can`                    | `isAuthenticated`, `hasPermission`, `canEdit` |
| Загрузка            | `load`                                  | `loadUserList`, `loadOrderDetail`             |
| Обработчик события  | `on`                                    | `onDelete`, `onFormSubmit`, `onRowClick`      |

Обработчики событий всегда начинаются с `on`.

## 5) Signals и RxJS

- По возможности используйте `signal`.
- Для подписок на `Observable` используйте `takeUntilDestroyed(...)`.

```typescript
@Injectable()
export class UserListFacade {
  private readonly userApiService = inject(UserApiService);
  private readonly destroyRef = inject(DestroyRef);

  public readonly userList = signal<UserModel[]>([]);
  public readonly isLoading = signal(false);
  public readonly error = signal<string | null>(null);

  public loadUserList(): void {
    this.isLoading.set(true);
    this.error.set(null);

    this.userApiService
      .getUserList()
      .pipe(takeUntilDestroyed(this.destroyRef))
      .subscribe({
        next: (apiDataList) => {
          this.userList.set(convertUserApiDataListToUserModelList(apiDataList));
        },
        error: (err: Error) => {
          this.error.set(err.message);
          this.isLoading.set(false);
        },
        complete: () => {
          this.isLoading.set(false);
        },
      });
  }
}
```

## 6) Конвертеры

Конвертеры должны быть чистыми функциями.

**Паттерны нейминга:**

- `convert<Entity>ToApi`
- `convert<Entity>ToClient`
- Для массивов: `...List...List`

```typescript
export const convertUserApiDataToUserModel = (apiData: UserApiData): UserModel => {
  return {
    id: apiData.id,
    firstName: apiData.first_name,
    lastName: apiData.last_name,
    createdAt: new Date(apiData.created_at),
  };
};

export const convertUserModelToUserApiData = (model: UserModel): UserApiData => {
  return {
    id: model.id,
    first_name: model.firstName,
    last_name: model.lastName,
    created_at: model.createdAt.toISOString(),
  };
};
```

## 7) Фасады

- Компонент не работает напрямую с API-сервисами.
- Фасад агрегирует API, форму, состояние и бизнес-логику.
- Нейминг: `сущность/фича + Facade` (`UserListFacade`, `UserFormFacade`).

> Примечание: `input()` и `output()` использовать можно и нужно для компонентной коммуникации. Фасад не является их заменой, а отвечает за orchestration и state-management на уровне фичи.

## 8) Суффиксы коллекций

Для массивов всегда используйте суффикс `List`.

| Переменная         | Тип           |
| ------------------ | ------------- |
| `user`             | `UserModel`   |
| `userList`         | `UserModel[]` |
| `filteredUserList` | `UserModel[]` |
| `itemList`         | `Item[]`      |

## 9) API-данные и модели

- API-модели: `snake_case`, суффиксы `ApiData`, `ApiResponse`.
- Клиентские модели: `camelCase`, суффикс `Model`.

```typescript
export interface UserApiData {
  id: number;
  first_name: string;
  last_name: string;
  created_at: string;
}

export interface UserModel {
  id: number;
  firstName: string;
  lastName: string;
  createdAt: Date;
}
```

## 10) Константы

Константы именуются в `UPPER_SNAKE_CASE`.

| Префикс      | Назначение             | Пример                                      |
| ------------ | ---------------------- | ------------------------------------------- |
| `MAX_`       | Верхние границы        | `MAX_RETRIES`, `MAX_FILE_SIZE`              |
| `DEFAULT_`   | Значения по умолчанию  | `DEFAULT_PAGE_SIZE`, `DEFAULT_LANGUAGE`     |
| Без префикса | Конфигурационные ключи | `API_URL`, `CACHE_TTL`, `SUPPORTED_LOCALES` |

## 11) Тестовые данные: Fixture и Mock

| Тип       | Когда использовать                     | Пример                     |
| --------- | -------------------------------------- | -------------------------- |
| `Fixture` | Нужны готовые, переиспользуемые данные | `user.fixture.ts`          |
| `Mock`    | Нужно подменить поведение зависимости  | `user-api-service.mock.ts` |

## 12) Чек-лист перед именованием

1. Это компонент? → `сущность-тип` (`user-list`, `user-item`, `user-form`).
   Если это route-level страница, она лежит в `ui/components/<page-name>/`.
2. Это API-сервис? → `сущность + ApiService` (`UserApiService`).
3. Это сервис бизнес-логики? → `роль + Service` (`AuthService`).
4. Это form-сервис? → `сущность + FormService` (`UserFormService`).
5. Это фасад? → `сущность/фича + Facade` (`UserListFacade`).
6. Это конвертер? → `convert...To...` (`convertUserModelToUserApiData`).
7. Это API-интерфейс? → `...ApiData` / `...ApiResponse`.
8. Это клиентская модель? → `...Model`.
9. Это массив? → суффикс `List`.
10. Это `FormControl`? → `xxxFormControl`.
11. Это `FormGroup`? → `xxxFormGroup`.
12. Это `FormArray`? → `xxxFormArray`.
13. Это обработчик события? → `on + действие`.
14. Это константа? → `UPPER_SNAKE_CASE`.
15. Для реактивного state применяются `signal` и корректное завершение подписок через `takeUntilDestroyed(...)`.
