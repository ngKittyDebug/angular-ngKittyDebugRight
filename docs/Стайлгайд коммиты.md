# Стайлгайд по коммитам

## Цель

Этот гайд основан на правилах оформления коммитов из Angular `DEVELOPERS.md` и адаптирован под текущее репо.

Его задачи:

- сделать историю коммитов читаемой;
- упростить ревью и поиск изменений;
- привести сообщения коммитов к формату, который проходит локальные проверки `commitlint`.

## Базовый принцип

Один коммит должен отражать одну логическую единицу изменений.

Хороший коммит отвечает на два вопроса:

- что изменено;
- зачем это сделано.

Если в одном коммите смешаны рефакторинг, исправление бага и документация, такой коммит нужно разбить.

## Формат commit message

Сообщение коммита состоит из `header`, опционального `body` и опционального `footer`.

```text
<type>(<scope>): <subject>

<body>

<footer>
```

### Header

`header` обязателен.

Формат:

```text
<type>(<scope>): <subject>
```

Где:

- `type` - тип изменения;
- `scope` - область изменения, опциональна;
- `subject` - краткое описание.

## Ограничения по оформлению

- Любая строка сообщения коммита не должна быть длиннее `100` символов.
- `subject` пишется в повелительном наклонении: `add`, `fix`, `update`, `remove`.
- Не начинайте `subject` с заглавной буквы.
- Не ставьте точку в конце `subject`.
- `body` и `footer` отделяются пустой строкой.

## Допустимые типы

В текущем проекте `commitlint` разрешает следующие типы:

- `build` - изменения сборки или зависимостей сборки;
- `ci` - изменения CI/CD, workflow, automation;
- `docs` - изменения только в документации;
- `feat` - новая функциональность;
- `fix` - исправление дефекта;
- `perf` - оптимизация производительности;
- `refactor` - рефакторинг без новой функциональности и без фикса бага;
- `revert` - откат предыдущего коммита;
- `style` - форматирование и stylistic-правки без изменения поведения;
- `test` - добавление или исправление тестов;
- `chore` - служебные изменения инфраструктуры и tooling, не попавшие в категории выше.

## Scope

`scope` опционален, но рекомендуется, если он помогает быстро понять область изменений.

В качестве `scope` используйте:

- имя фичи: `main`, `auth`, `space`;
- имя слоя: `api`, `ui`, `router`;
- имя инструмента: `eslint`, `netlify`, `github-actions`.

Примеры:

- `feat(space): add NASA photo request flow`
- `fix(router): handle unknown routes`
- `ci(netlify): update deploy configuration`

Если изменение затрагивает несколько областей и один `scope` только запутает, его можно опустить:

```text
docs: update contribution guidelines
```

## Subject

`subject` должен быть коротким и конкретным.

Хорошо:

- `fix(space): handle empty photo response`
- `docs(commits): add commit message examples`
- `ci(badges): generate vitest coverage badge`

Плохо:

- `fix(space): Fixed bug with photos.`
- `feat: Added a lot of new functionality`
- `docs: changes`

## Body

`body` нужен, когда одного `subject` недостаточно.

В `body` стоит писать:

- мотивацию изменения;
- предыдущее поведение;
- новое поведение;
- важные ограничения или побочные эффекты.

Пример:

```text
fix(space): handle empty photo response

NASA API can return an empty payload for some dates.
Before this change the facade tried to read fields from undefined and crashed.
Now the service returns a safe fallback model and the page stays interactive.
```

## Footer

`footer` используется для:

- `BREAKING CHANGE`;
- ссылок на задачи и issue;
- служебных пояснений.

Пример с issue:

```text
fix(space): prevent duplicate photo requests

Closes #42
```

Пример с breaking change:

```text
refactor(api): rename space photo response model

BREAKING CHANGE: SpacePhotoModel.date is renamed to requestedAt
```

## Revert-коммиты

Для отката используйте формат:

```text
revert: <header исходного коммита>
```

В `body` указывайте:

```text
This reverts commit <hash>.
```

Пример:

```text
revert: feat(space): add photo loading facade

This reverts commit 1a2b3c4d.
```

## Практические рекомендации

- Делайте небольшие коммиты, которые можно понять без чтения всей ветки.
- Не используйте расплывчатые сообщения вроде `fix`, `update`, `changes`, `wip`.
- Не смешивайте механические правки и смысловые изменения в одном коммите.
- Если добавляете новую функциональность и сразу исправляете сопутствующий баг, чаще всего это два разных коммита.
- Перед пушем перечитайте историю ветки: она должна читаться как короткий рассказ о работе.

## Примеры для этого проекта

```text
feat(main): add layout shell for application routes
fix(space): return fallback image on API error
docs(pr): update draft PR checklist
test(space): add facade coverage for empty response
ci(badges): switch coverage badge to vitest label
build(deps): update Angular CLI packages
chore(husky): enable commit message validation
```

## Когда body обязателен

Рекомендуется писать `body`, если:

- из `subject` неочевидно, зачем сделано изменение;
- изменение меняет поведение существующего кода;
- есть компромисс, ограничение или временное решение;
- меняется инфраструктура, CI или deployment;
- коммит будет сложно понять через неделю без дополнительного контекста.

## Что проверять перед коммитом

- Сообщение соответствует формату `<type>(<scope>): <subject>`.
- Выбран корректный `type`.
- `subject` короткий, в повелительном наклонении и без точки.
- Длина строк не превышает `100` символов.
- Один коммит содержит одну логическую единицу изменений.

## Связанные правила проекта

- Имена веток в проекте валидируются отдельно и должны соответствовать шаблону:
  `^(chore|feat|fix|docs|style|refactor|perf)\/[a-zA-Z0-9-]+[_-][a-zA-Z0-9-]+$`
- Оформление PR описано отдельно в документе `docs/Стайлгайд PR.md`.

## Короткий шаблон

```text
<type>(<scope>): <subject>

<why / previous behavior / new behavior>

<issue reference or BREAKING CHANGE>
```
