# Стайлгайд по оформлению Pull Request

## Цель

Сделать процесс ревью предсказуемым и быстрым: пока работа в процессе, PR должен быть в статусе `Draft`, а в `Ready for review` переводится только после самостоятельной проверки изменений.

## Жизненный цикл PR

### 1) Первый коммит -> сразу `Draft PR`

- Создавайте ветку от GitHub Issue на доске (через создание branch из карточки задачи).
- Базовой веткой выбирайте актуальную целевую ветку проекта.
- После первого коммита сразу открывайте PR в статусе `Draft`.
- Не откладывайте создание PR до конца задачи: ранний Draft помогает видеть прогресс и контекст.
- В заголовке PR используйте номер задачи чтобы была явная связь с задачей.

### 2) Работа в `Draft`

- Коммитьте небольшими, логически завершенными шагами.
- Обновляйте описание PR, если меняется объем или подход.
- Если есть промежуточные технические решения, фиксируйте их в описании.
- Не запрашивайте ревью, пока не пройден self-check.

### 3) Перевод в `Ready for review`

Переводите PR в `Ready for review` только когда:

- реализованы все пункты задачи;
- код локально собирается;
- тесты и проверки, относящиеся к изменению, проходят;
- сделан self-check по чеклисту ниже;
- в PR нет временных/черновых решений.

## Self-check перед `Ready for review`

- Код собирается, тесты проходят.
- Проверены принципы SOLID.
- Проверены code smells (дублирование, слишком сложные методы, магические значения, лишние зависимости).
- Дописана документация к методам, где это требуется.
- Внимательно просмотрен diff: нет временного кода, debug-фрагментов и случайных изменений.
- В PR указан номер задачи.

## Требования к содержимому PR

- Понятный заголовок: что сделано и для чего.
- В описании:
  - ссылка на задачу;
  - краткий контекст и цель изменений;
  - что именно проверять ревьюеру;
  - шаги проверки (test plan).
- Не смешивайте в одном PR несвязанные изменения.
- Если изменения большие, разбивайте на несколько PR.
- Если реализуется большая фича, декомпозируйте ее на связанные задачи на доске и создавайте маленькие PR в одну общую feature-ветку.

## Базовые рекомендации по коммитам

- Один коммит = одна логическая единица изменений.
- Сообщение коммита отвечает на вопрос "зачем это изменение".
- Перед финальным пушем проверьте историю коммитов на читаемость.

## Когда не переводить в `Ready for review`

- Есть незавершенные части задачи.
- Не пройдены тесты или сборка.
- В diff остались временные правки.
- Описание PR не заполнено и не содержит ссылки на задачу.

## Итоговое правило

`Draft` с первого коммита -> `Ready for review` только после полного self-check.
